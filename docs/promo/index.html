Tessera is a personal agentic coding framework for Claude Code — forked, pruned, and rebuilt around what survives real dogfooding instead of what looks good in a README. Verification gates, a YAGNI discipline, and an honest record of every decision, including the ones that turned out wrong.

What it actually is

Not a SaaS product, not a plugin marketplace. A working framework that governs how Claude Code behaves on real projects — skills, hooks, rules, and a written trail of every consequential decision.

Origin

Forked & pruned

Started from maggy, stripped to what's useful for solo dogfooding, then rebuilt around clean extension points — project profiles, an ADR system, an honest override mechanism.

Mechanism

Suggested, not silent

Heavy machinery — loops, multi-agent pipelines, multi-engine review — is proposed by Claude and accepted, declined, or modified by the human. Auto-invocation is rejected as a default pattern.

Discipline

Verifiable over ceremonial

A TDD Stop hook that actually runs tests beats a ten-stage agent pipeline nobody reads. Reviewers operate read-only. Decisions get written down, not just remembered.

Four influences, deliberately balanced

Tessera doesn't pick one philosophy of agentic coding and run with it. It takes the part of each that survives contact with real work.

Structure compounds — selectively true

Keep the TDD Stop hook, the read-only reviewer pipeline, the conditional-rules layer. Reject the maximalist scaffolding. Maggy proved that infrastructure (typed memory graphs, three-layer compaction defense, auto-fresh code graphs) can be genuinely sophisticated — but a ten-stage pipeline for solo work is ceremony, not structure.

Master the primitives

Short CLAUDE.md, mistake-driven rules, native commands first. This is the dominant philosophy for the dogfood phase: don't reach for a heavier custom thing if a native Claude Code feature already covers the gap.

The YAGNI ladder

Climb the smallest-thing-that-works ladder before reaching for the heavier abstraction — stdlib, then native platform, then an installed dependency, then one line, only then custom code. Applies to code and to tooling decisions about Tessera itself.

Ensembling is a tool, not a default

Multi-model review and consequential-decision arbitration are real, valuable patterns for discrete high-stakes calls — and overhyped as a default for everyday agentic coding. Reserved for decisions that actually warrant it.

The skill mosaic

Four skills load eagerly into every session. Everything else activates on demand — by file path, explicit invocation, or context. Skill defaults are starting points, not prescriptions (principle 15).

Project profiles

A project declares its profile in .tessera/project.yml. Non-matching projects pay none of the overhead. Going up a tier is frictionless; going down warns and asks for confirmation.

Decisions, written down

Architecture Decision Records are numbered, dated, and immutable once accepted — superseded by a new ADR rather than edited. The Observatory tracks everything noticed but not yet decided: lighter than an ADR, a junk drawer with structure.

What dogfooding actually found

Tessera's own development is governed by Tessera. That produces an unusually honest trail — including the bugs in its own memory layer and the conventions that turned out not to work.

"A kill/keep test on observed output is only valid once the input path is verified live. 'Feature never fired' must be disambiguated — unused vs. unreachable — before it counts as evidence."
